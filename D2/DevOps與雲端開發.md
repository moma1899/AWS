## 一、AWS IAM 開發人員群組（DEVELOP）與 S3 權限配置

### 第一階段：AWS 雲端主控台 IAM 群組授權操作流程

1. **進入權限管理中心：**
   登入 AWS 管理主控台，在上方搜尋列輸入並進入：**IAM**

2. **鎖定使用者群組：**
   點擊左側導覽選單中的 **User groups**（使用者群組），在清單中點選開發團隊群組名稱：`DEVELOP`

3. **發起權限調整：**
   進入群組詳細頁面後，點擊右側的 **Add permissions** 按鈕，並從下拉選單中選取 **Attach policies directly**（直接附加政策）。

4. **篩選與勾選 S3 官方政策：**
   在權限政策搜尋欄位中輸入關鍵字：`AmazonS3FullAccess`，在下方搜尋結果列表中，準確**勾選** AmazonS3FullAccess 這一列項目。

5. **確認變更並生效：**
   檢視無誤後，點擊右下角的 **Add permissions** 藍色按鈕。此時系統會跳出授權成功提示，權限立刻全面生效。

### 第二階段：群組成員權限繼承與自動生效驗證

**權限自動繼承機制：**

當上述設定完成後，所有被歸類在 **DEVELOP** 群組底下的 IAM 使用者帳號（例如 `DEV_moma1899`），完全不需要單獨進行任何手動設定，皆會自動全面繼承該群組的所有雲端權限。

**群組成員自動解鎖的 S3 核心操作能力：**

| 權限類型 | 說明 |
|---------|------|
| **建立權限** | 允許自由建立全新的 S3 Bucket（儲存桶） |
| **上傳權限** | 允許將本機檔案、前端靜態網頁資產上傳至指定的儲存桶中 |
| **刪除權限** | 允許清理、移除儲存桶內不再需要的過期檔案 |
| **下載權限** | 允許讀取與下載儲存桶內的所有資料物件 |

> **終極狀態：** 帳號完整擁有 **S3 Full Access** 的最高控制權。

### 第三階段：企業團隊 IAM 權限架構對照表

以下為目前專案團隊在 AWS 雲端環境上的角色權限劃分與階層架構：

| 群組名稱（Group） | 附加權限政策（Policy） | 涵蓋目標與適用對象 |
|-----------------|---------------------|-----------------|
| **ADMIN** | AdministratorAccess | 雲端最高管理者，擁有全服務的最高變更與控制權 |
| **DEVELOP** | AmazonS3FullAccess | 團隊開發人員（如 DEV_moma1899），專注於 S3 儲存桶的完整配置與檔案讀寫 |

## 三、AWS CLI 建立 S3 儲存桶與基礎檔案操作指南

### 第一階段：建立全新 S3 儲存桶（Bucket）

- **操作目的：** 在 AWS 雲端上開闢一個全新且獨立的線上儲存空間。

```bash
aws s3 mb s3://ck101-22
```

> **系統成功回傳：** `make_bucket: ck101-22`

### 第二階段：檢查雲端儲存桶列表

- **操作目的：** 確認剛剛建立的儲存桶是否已經成功同步存在於你的 AWS 帳號清單中。

```bash
aws s3 ls
```

> **系統成功回傳：** 畫面上會列出你目前擁有的所有儲存桶，其中必定會精準看到 `ck101-22`。

### 第三階段：上傳測試檔案至儲存桶

- **操作目的：** 實測本地端憑證對該儲存桶是否真正擁有寫入與上傳的權限。

```bash
aws s3 cp test.txt s3://ck101-22/
```

> ⚠️ **備註說明：** 執行此指令前，請確保你當前終端機所在的資料夾目錄下，確實存在 `test.txt` 這個測試文字檔。

### 第四階段：檢查儲存桶內部檔案清單

- **操作目的：** 深入查看指定儲存桶內部目前存放了哪些物件，用來雙重驗證剛才的檔案是否有成功推送上去。

```bash
aws s3 ls s3://ck101-22
```

> **系統成功回傳：** 畫面上會吐出該物件的上傳日期、時間、檔案大小，並清清楚楚列出：`test.txt`

## 五、Docker Image 打包與 Docker Hub 雲端推送

### 第一階段：優化 Dockerfile 核心基底

**調整策略：** 將基礎映像檔（Base Image）改用官方的 `python:3.11-slim` 版本。

**改進優勢：**
- 相較於完整版 Python 映像檔，`slim` 版本大幅精簡了不必要的系統套件
- 縮小高達 60% 以上的檔案體積
- 加快網路傳輸速度
- 減少潛在的資安漏洞風險

```dockerfile
FROM python:3.11-slim
```

### 第二階段：本地端編譯與封裝映像檔（Build）

- **操作目的：** 在本地端將專案原始碼、環境變數與套件規格，打包成一個名為 `aether-panel` 的獨立靜態映像檔。

```bash
docker build -t moma1899/aether-panel:latest .
```

### 第三階段：Docker Hub 雲端倉庫登入驗證（Login）

- **操作目的：** 透過終端機向 Docker Hub 全球伺服器進行身分驗證，開通上傳權限。

```bash
docker login <USERNAME>
```

按下 Enter 鍵後，畫面上會跳出密碼輸入提示，請手動貼上你的 Docker Hub 密碼或個人存取權杖（Access Token）。

> **成功指標：** 當畫面上顯示 `Login Succeeded` 時即代表密鑰對接成功。

### 第四階段：正式推送映像檔至雲端倉庫（Push）

- **操作目的：** 將本地編好的映像檔正式推送到雲端倉庫儲存。

```bash
docker push moma1899/aether-panel:latest
```

> **上線驗證指標：** 執行後終端機會開始跑進度條，將各個層級（Layers）的資料上傳。上傳完成後，直接登入 Docker Hub 網頁控制台，就會看到倉庫中已經安穩地躺著剛剛推送上去的最新映像檔。

> **最終效益：** 未來在 AWS EC2 虛擬電腦上，完全不需要複製任何程式碼原始檔，只要直接執行 `docker pull` 就能一鍵下載並立即運行！

## 七、AWS EC2 綁定 IAM Role 與 MobaXterm 驗證 S3 權限安全

### 第一階段：建立雲端專屬 IAM Role 角色

1. **安全身分建置路徑：**
   ```
   登入 AWS 管理主控台 ➔ 進入 IAM 服務 ➔ 點選左側選單 Roles（角色） ➔ 點擊右上角 Create role（建立角色）
   ```

2. **信賴實體類型選擇：** 選取 **AWS service**（AWS 服務），並在下方指定授權服務為 **EC2**，這代表允許未來的 EC2 虛擬主機直接代表你執行 AWS 內部命令。

3. **給予全新角色名稱：** 在名稱欄位輸入自訂角色標籤：`custom_role`

4. **附加 S3 完整操作政策：** 在權限搜尋欄輸入並勾選官方核心政策：`AmazonS3FullAccess`，確認無誤後點擊下方的 **Create role**。

> **建置完成指標：** 控制台上方跳出「角色 custom_role 已建立」的綠色成功提示，即代表該角色的雲端設定大功告成。

### 第二階段：將 IAM Role 部署掛載至 EC2 主機

1. **鎖定目標虛擬電腦：**
   ```
   前往 EC2 主控台 ➔ 點擊左側「執行個體」(Instances) ➔ 在清單中勾選正在運作的 Linux 主機
   ```

2. **啟動身分修改程序：**
   ```
   點選右上角的「動作」(Actions) ➔ 展開「安全性」(Security) ➔ 點選「修改 IAM 角色」(Modify IAM role)
   ```

3. **綁定並完成更新：** 在下拉選單中精準選取剛剛建立的 `custom_role`，接著點擊右下角的 **Update IAM role**，權限即刻全面生效。

### 第三階段：使用 MobaXterm 遠端登入與 CLI 權限實測

**發起安全 SSH 連線：** 打開 MobaXterm ➔ 點選 Session ➔ 選擇 SSH，並依序填入以下連線通訊參數：

| 欄位 | 輸入值 |
|------|-------|
| 遠端主機 IP（Remote host） | `43.213.29.254` |
| 系統預設帳號（User） | `ec2-user` |
| 專屬私密金鑰（PEM） | 載入 `ck101-linux.pem` |

> **成功穿透提示：** 當畫面上順利跳出命令提示字元 `[ec2-user@ip-172-31-28-62 ~]$` 時，代表已安全連線進入 EC2 Linux。

**在 Linux 命令列直接執行儲存桶列表指令：**

```bash
aws s3 ls
```

> **連線成功驗證：** 畫面上會直接吐出目前帳號下的儲存桶資產，例如：`2026-05-28 08:21:00 ck101-22`。主機在完全沒有設定任何永久金鑰的情況下成功讀取 S3，代表 IAM Role 權限已完美接通。

### 第四階段：企業正式環境資安鐵律（為什麼要用 IAM Role？）

**內部自動安全交換機制：**

在正式生產環境中，EC2 主機會透過綁定的 IAM Role 角色，在背後自動向 AWS 官方安全權杖服務請求**過期即立刻失效的暫時憑證**（Temporary Credentials）。

**杜絕金鑰外流慘劇：**

採用此方案可以徹底避免將 Access Key ID 與 Secret Access Key 直接寫死在專案原始碼中。即使未來專案程式碼不小心被推上公開的 GitHub 倉庫，駭客也完全拿不到任何雲端通行證，直接斬斷金鑰外流、帳單被駭客刷爆的資安隱患。

### 第五階段：Flask 搭配 Boto3 的高階免金鑰寫法

**Python 原始碼潔淨配置：** 後端在撰寫 `feature3` 的檔案管理功能時，引用套件並宣告連線的程式碼只需要簡單兩行：

```python
import boto3
s3 = boto3.client("s3")
```

**底層自動偵測機制：** 當這段程式碼在掛有角色的 EC2 電腦上執行時，AWS 官方 SDK（boto3）會自動向主機環境探查 IAM Role 憑證並自動通過 AWS 身分認證。

> ⚠️ **嚴禁出現的程式碼：** 任何形式的硬編碼金鑰（如 `aws_access_key_id="AKIAXXXXX"`）都絕對不能出現在程式碼中。

### 第六階段：本地到雲端免金鑰操作完整生命週期

以下為權限架構與程式碼運作的無縫對接完整流向：

        ▼

        │
        │  維運管理連線
        ▼

        │
        │  環境功能測試

        │

        ▼
